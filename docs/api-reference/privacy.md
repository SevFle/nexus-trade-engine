# Privacy (GDPR / CCPA)

Data-subject request lifecycle: export, deletion, rectification,
restriction, objection. Source:
[`engine/api/routes/privacy.py`](../../engine/api/routes/privacy.py),
[`engine/privacy/`](../../engine/privacy/).

The engine records one `dsr_requests` row per request for SLA
audit (GDPR Art. 12 mandates a 1-month response window; the row's
`sla_due_at` defaults to 30 days from creation).

## Endpoints

### `POST /api/v1/privacy/export`

Initiate a GDPR data export. Returns a job id; the actual bundle
is generated by a worker (see
[`engine/privacy/export.py`](../../engine/privacy/export.py)) and
either emailed or staged for download depending on operator
config.

**Request body** (optional):

```json
{ "kinds": ["user", "portfolios", "orders", "backtests"] }
```

Defaults to all kinds. The exported bundle is a JSON document
plus per-section NDJSON files; blobs (PDF tax reports, etc.) are
inlined as base64.

**Response** `202 Accepted`:

```json
{
  "request_id": "<uuid>",
  "status": "pending",
  "sla_due_at": "2026-07-05T12:00:00Z"
}
```

### `POST /api/v1/privacy/delete`

Initiate a deletion request. The caller's account is **not**
deleted immediately — a 14-day grace period lets the user cancel
(see [cancel endpoint](#post-apiv1privacydeletecancel)).

**Response** `202 Accepted`:

```json
{
  "request_id": "<uuid>",
  "status": "pending",
  "sla_due_at": "2026-07-05T12:00:00Z",
  "effective_at": "2026-06-19T12:00:00Z"
}
```

After `effective_at`, the worker hard-deletes the user row and
cascades. `legal_acceptances` rows are RETRICT'd — they survive
the delete to preserve the audit trail (this is a deliberate
constraint, see [data model](../architecture/database.md)).

### `POST /api/v1/privacy/delete/cancel`

Cancel a pending deletion request. No-op if already cancelled or
completed.

### `GET /api/v1/privacy/delete/status`

Return the caller's current deletion request, if any.

**Response** `200 OK` — `DeletionStatusResponse`:

```json
{
  "request_id": "<uuid>",
  "status": "pending",
  "effective_at": "2026-06-19T12:00:00Z",
  "cancelled_at": null
}
```

### `GET /api/v1/privacy/requests`

List all DSR requests the caller has filed (export, delete,
rectify, restrict, object). Sorted newest-first.

**Response** `200 OK` — `DSRListResponse`.

### `GET /api/v1/privacy/kinds`

Return the supported `kind` values (for client-side form
rendering). Publicly stable; safe to call without auth.

## Operator workflow

Operators are responsible for actually serving the bundle to the
user. The engine writes a `dsr_requests` row + the worker
generates the export file; delivery (email, signed download URL,
postal mail) is out of scope. See the [legal/processors
template](../legal/processors.md) for the sub-processor inventory
the operator must maintain.

## Implementation notes

- The exporter uses an outer-join against `backtest_results` so
  orphan rows (where `portfolio_id` was deleted before the user)
  are still included. gh#157 fixed an earlier bug where they were
  dropped silently.
- Deletion is hard, not soft, once it executes. There is no
  undelete.
- `rectify` / `restrict` / `object` kinds write the request row
  but do not have automated worker actions yet — operators handle
  them out-of-band and update the row's `status` and
  `completed_at` when done.
